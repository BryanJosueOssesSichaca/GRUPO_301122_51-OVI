Crear y sincronizar repositorio Remoto en Github

Vincular un repositorio Local con el repositorio remoto en Github (Git Bash)

Si se esta trabajando con la linea de comandos lo primero que se debe realizar es crear un repositorio desde la página oficial de GitHub.

Se da clic sobre el botón Start a project en la página principal de GitHub, la inetrfaz web permite crear un nuevo repositorio.

  1. Owner: la cuenta de Github a la que estará vinculado el repositorio.
  2. Repository Name: para el nombre que tendrá el repositorio.
  3. Description: una descripción opcional del respositorio.
  4. Public/private indicar si el repositorio será público o privado (Para crear repositorios privados en GitHub se debe contar con una cuenta Enterprise).
  5. Initialize this repository with a README:Crear un archivo README.md junto con el repositorio.
  6. Add gitIgnore/License: agregar licenciamiento al código fuente del repositorio (en caso de ser un repositorio público, solo estarán disponibles licencias de software libre y Open Source), así como el archivo gitignore.

Creando repositorio en Github (08 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

Una vez que este creado el repositorio se da clic sobre el botón Clone or download y se copia la URL del repositorio remoto en GitHub.

Boton Clone or Download - Github (08 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

Al tener la URL del repositorio remoto en GitHub, se dirige a la linea de comandos de Git (Git bash), se indica la ruta del repositorio local creado (o crear uno nuevo, el nombre del repositorio local puede ser diferente al del remoto) y se introduce el comando remote:

$ git remote add origin URL-repositorio-remoto-Github

De esta manera el repositorio local estará vinculado con el repositorio remoto en GitHub.

Si se desea visualizar el repositorio remoto vinculado se indica el parámetro -v al comando remote:

$ git remote -v

Si se desea eliminar el repositorio remoto que tenemos vinculado se indica el parámetro remove al comando remote

$ git remote remove origin

Vincular el repositorio Local con el repositorio remoto en GitHub (Github Desktop)

Una vez creado el repositorio local, se da clic sobre el botón publish repository, se abrirá una nueva ventana donde se coloca el nombre del repositorio remoto, la descripción y se indica si será publico o privado (Para crear repositorios privados en GitHub se debe contar con una cuenta Enterprise).

Automaticamente GitHub Desktop creará un repositorio remoto con la cuenta de GitHub con la que se ha iniciado sesión en GitHub Desktop y se sincronizará con el repositorio local.

Fetch, Push y Pull

Fetch, Push y Pull (Git Bash)

Cuando el repositorio local esta vinculado con un repositorio remoto, es muy importante que antes de empezar a trabajar o hacer cambios en el proyecto se debe tener la última versión con todos los commits del repositorio remoto, para esto se utiliza el comando fetch

$ git fetch: origin

El comando fetch recupera todos los datos del repositorio remoto y en caso de existir nuevos cambios, informará sobre estos.

comando fetch - Git Bash (08 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

Es muy importante no olvidar hacer fetch antes de comenzar a realizar cambios en el repositorio local, porque si se intenta publicar los cambios del repositorio local con una versión distinta a la del repositorio remoto, Git no lo permitirá.

comando fetch error - Git Bash (08 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

El comando fetch unicamente recupera la información pero no la une con la que se encuentra en el repositorio local, para actualizar el repositorio local con los cambios que puedan existir en el repositorio remoto se utiliza el comando pull

$ git pull: origin Nombre_rama

En caso tal de no tener la misma versión en el repositorio local, por defecto Git no permitirá unir los cambios (fatal: refusing to merge unrelated histories).

Para solucionar este problema se indica al comando pull el parámetro --allow-unrelated-histories.

$ git pull: origin Nombre_rama --allow-unrelated-histories

Al utilizar el comando pull se abrirá el editor (Por defecto Git bash utiliza el editor vim) configurado y se deberá escribir la descripción del commit con la mezcla de los cambios en el repositorio remoto con el local.

comando pull - Git Bash (08 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

En este punto se realizarán los cambios al proyecto de manera local, cuando se hayan confirmado todos los cambios a través de commits en el repositorio local, se deberán enviar al repositorio remoto.

Para esto se utiliza el comando push

$ git push: origin Nombre_rama

Git solicitará el nombre de usuario de Github y la contraseña para poder confirmar los cambios en el repositorio remoto cada vez que se utilíce el comando push, en caso de que los datos de inicio de sesión de GitHub sean erróneos, Git no permitirá mezclar los cambios locales con el repositorio remoto.

comando push - Git Bash (08 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

Fetch, Push y Pull (Github Desktop)

Para hacer fetch en GitHub Desktop se da clic sobre el botón Fetch Origin

En caso de existir cambios en el repositorio remoto que no estén en el repositorio local, el botón fetch se sustituirá por el botón pull origin y un número que indica la cantidad de commits nuevos, de esta manera se hace pull y se traen los cambios del repositorio remoto al local.

Captura Botón pull Origin - Github Desktop (08 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

Ahora, se trabaja de manera local y cuando al confirmar los cambios a través de commits, aparecerá el botón push origin y un número indicando la cantidad de commits realizados de manera local que seran enviados al repositorio remoto.

Publicando Proyecto en Github Pages

La plataforma de GitHub ofrece un servicio de Hosting para alojar los repositorios en una página web de manera muy sencilla y sin necesidad de tener conocimientos en administración de servidores o clientes FTP.

Este servicio es denominado Github Pages.

Para poder publicar el repositorio en Github pages, la rama que será publicada debe contener un archivo index.html. de lo contrario el servidor arrojará un mensaje de error 404 Indicando que no se encuentra el archivo index.html.

Error 404 - Github Pages (08 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

Se debe dirigir a la opción settings en la pagina de Github del repositorio remoto.

settings - github (08 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

Se dirige al apartado Github Pages y se da clic sobre la lista desplegable Source, por defecto estará seleccionada la opcion None.

Se abrirá la lista desplegable y se mostrarán las ramas que se tienen en el proyecto, para este caso solo existe la rama Master, por lo tando dentro de Master debe existir un archivo index.html, se selecciona la rama (en el ejemplo Master branch) y se da clic sobre el botón save.

Aparecerá un mensaje indicando que el sitio web esta listo para ser publicado, se recarga la página y el mensaje cambiará a Your site is published.

settings - github (08 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

Nota: también es posible publicar proyectos en Github Pages creando una rama con el nombre gh-pages y publicando dentro de la misma el archivo index.html.

URL del proyecto publicado en Github Pages

Github pages permite modificar el dominio pero por defecto la URL de un proyecto publicado en Github pages se organiza de la siguiente manera:

https://Nombre_De_Usuario_Github.github.io/Nombre_del_repositorio_Remoto/
