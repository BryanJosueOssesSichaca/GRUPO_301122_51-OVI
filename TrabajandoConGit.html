Flujo de Trabajo

En GIT existen 3 diferentes estados con los que trabajamos

1 Working directory

Aquí editamos, agregamos los diferentes archivos y trabajamos en nuestro proyecto.

2 Staging área

Aquí es donde escogemos que archivos están listos para pasar al tercer estado, al igual que decidimos que archivos no están listos.

3 repository

Aquí se lleva todo el registro de nuestro proyecto (una vez hacemos commit).

Flujo de trabajo en Git (06 de Abril del 2018) “elaboración propia”

por lo tanto (1) siempre tendremos un directorio en el cual agregaremos o modificaremos archivos, una vez que la modificación de estos archivos este lista debemos de seleccionar que archivos estan listos y colocarlos en la staging area de esta manera indicamos que cambios queremos incluir en el (3) commit que llevara registro de los cambios de nuestro proyecto.

Preparando archivos para confirmación

Agregar archivos al staging área (Git Bash)

Una vez que se han producido cambios en los archivos existentes o se han agregado nuevos archivos al directorio de trabajo, si utilizamos el comando $ git status podremos ver que nos indica que archivos han sido modificados o agregados, asi como cuales cambios estan listos para confirmar y cuales no.

Git Bash - comando status (07 de Abril del 2018) “elaboración propia”

Si queremos ver con exactitud, las líneas de código agregadas o eliminadas en estos cambios podemos utilizar el comando diff:

$ git diff

si queremos ver únicamente los cambios sobre los archivos que están en el staging área o que están preparados para confirmación tenemos que agregar el parámetro --staged

$ git diff --staged

una ves que ya sabemos que cambios se han realizados, para poder hacer commit sobre estos cambios, primero debemos añadir estos cambios al staging área, para eso utilizamos el comando add:

$ git add nombre_del_archivo

Si hemos creado o modificado multiples archivos podemos agregar todos estos cambios de una sola vez al staging área agregando el parámetro --A o --all

$ git add --all

NOTA: para quitar un archivo del staging area utilizamos el comando reset:

$ git reset HEAD nombre_del_archivo

Git Bash - comando reset, archivos preparados (07 de Abril del 2018) “elaboración propia”

Recordar que los archivos que son agregados en el commit (Confirmación) son aquellos que están en el staging área y han quedado registrados no aquellos archivos que solo han sido modificados.

Git Bash - comando status, archivos preparados (07 de Abril del 2018) “elaboración propia”

Para agregar un archivo al staging área hay que marcar el checkbox que se encuentra al lado izquierdo del nombre del archivo y viceversa para quitar un archivo del staging area.

Confirmando Cambios

Confirmando cambios (Git Bash)

Para confirmar los cambios debemos utilizar el comando commit (Confirmación):

$ git commit --m "Descripción del commit"

El parámetro --m sirve para escribir directamente la descripción del commit en la linea de comandos o terminal, en caso de no agregar el parámetro --m se abrirá el editor que hayamos configurado para trabajar con GIT o el editor por defecto del sistema con el fin de escribir una descripción para el commit o confirmación.

Commit Git bash (07 de Abril del 2018) “elaboración propia”

En caso de querer modificar el mensaje de la descripción del commit podemos utilizar el parámetro --amend

$ git commit --ammend

mientras nos encontremos en una rama local podemos deshacer el commit o confirmación realizada utilizando el comando reset

$ git reset --soft HEAD ~1

al agregar el parametro --soft eliminamos el commit pero nuestros archivos quedan intactos en caso de querer borrar todo lo que contenga el commit utilizar el parametro --hard

HEAD (cabeza) representa al commit en el cual nos encontramos, normalmente es el ultimo commit realizado , ~1 es la cantidad de commits que se retrocederan con respecto a HEAD.

En caso de querer Deshacer un commit que ya se haya subido a una rama remota debemos utilizar el comando revert

$ git revert HEAD

Podemos sustituir HEAD por el código SHA del commit, por si aún no has leído el artículo crear y clonar repositorios con Git, explicaba que para ver el historial de commits utilizábamos el comando log, el cual nos imprime en pantalla entre otros datos el código SHA de confirmación del commit.

Confirmando cambios (Github desktop)

en el panel principal de Github Desktop, en la pestaña changes encontramos dos campos de texto, el primero que es llamado summary y el segundo mas grande description. En summary se debe escribir el nombre del commit mientras que en description la descripcion del commit.

por ultimo debemos dar click en el boton llamado commit to "Master" (Master hace referencia a la rama maestra o principal del repositorio, mas adelante en este mismo articulo se hablara sobre las ramas).

Github Desktop, commit (07 de Abril del 2018) “elaboración propia”

en caso de querer revertir un commit, tenemos que ir a la pestaña History y dar click derecho sobre el commit que queramos eliminar, aparecera un menú emergente con la opcion revert this commit.

Ramas o bifurcaciones

una rama representa una linea del tiempo en nuestro proyecto, son caminos independientes que pueden evolucionar de manera distinta, en la practica representan diferentes estados que nuestro proyecto puede tener.

los cambios realizados en una rama solo se reflejan en la misma, es por esto que podemos experimentar o hacer grandes cambios en una rama diferente sin afectar la rama principal Master u otras ramas dentro del repositorio Git.

cuando creamos un repositorio, Git nos creará automaticamente una rama llamada Master, la rama por defecto de Git es la rama Master y apartir de esta misma es que se crearán nuevas ramas o bifurcaciones.

Ramas (Git Bash)

Si queremos ver las ramas que posee nuestro repositorio debemos utilizar el comando branch:

$ git branch

si queremos crear una nueva rama en nuestro proyecto debemos pasarle como parametro al comando branch el nombre de la rama.

$ git branch nombre_Rama

Git bash, creando nueva rama (07 de Abril del 2018) “elaboración propia”

para cambiar entre ramas debemos utilizar el comando checkout:

$ git checkout nombre_Rama

Git bash, moverse entre ramas (07 de Abril del 2018) “elaboración propia”

una ves que cambiemos de rama los cambios se realizarán sobre esta misma y no sobre master, si queremos reflejar los cambios que hemos realizado sobre esta nueva rama tendremos que fusionarlas (merge).

al hacer merge entre dos ramas se generará un nuevo commit con los cambios derivados de la fusion de esas dos ramas.

para fusionar ramas debemos utilizar el comando merge

$ git merge nombre_Rama

Git bash, mezclando ramas (07 de Abril del 2018) “elaboración propia”

primero debemos situarnos sobre la rama que va a absorver a la otra (utilizamos el comando checkout para movernos entre ramas), en el ejemplo me sitúo sobre la rama Master para poder absorver en Master la otra rama llamada nueva_rama.

Si queremos eliminar una rama local tenemos que pasarle el parametro -D al comando branch

$ git branch -D nombre_Rama

Git bash, borrando ramas (07 de Abril del 2018) “elaboración propia”

Ramas (Github Desktop)

Para crear ramas, en el panel principal de Github Desktop debemos ir a la pestaña llamada Current branch damos click y se desplegará un nuevo menú, en este nuevo menú en la pestaña branches hay un campo de texto en el cual debemos escribir el nombre de la nueva rama y finalmente dar click en el boton new branch para crear una nueva rama.

Github Desktop, creando ramas (07 de Abril del 2018) “elaboración propia”

Para cambiar de ramas, tenemos que dar click sobre el nombre de la rama en la que queremos situarnos, la rama en la que estamos situados es la que esta marcada con un icono dentro de la lista.

Github Desktop, moviendose entre ramas (07 de Abril del 2018) “elaboración propia”

para borrar una rama, primero tenemos que seleccionarla e ir al menú branch -> delete

Github Desktop, eliminando ramas (07 de Abril del 2018) “elaboración propia”
