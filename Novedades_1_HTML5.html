DOCTYPE


Esta declaración indica en que lenguaje se interpretara el código fuente en la página web.
La declaración doctype le ordena al navegador la versión del lenguaje HTML utilizado en la página. Se hace referencia a la marcación Document Type Definition (DTD). La DTD especifica las reglas usadas por el lenguaje de marcación para que así se pueda presentar el contenido de manera adecuada.
A continuación se relacionan los tipos de Doctype ocupados en HTML

Versión Características
HTML 4.01 estricto Permite todos los elementos y atributos HTML, pero no permite etiquetas de presentación, como la etiqueta de fuente. No se permiten conjuntos de marcos.
HTML 4.01 transicional Igual que HTML estricto, pero permite el uso de elementos en desuso, como la etiqueta de fuente. No se permiten conjuntos de marcos.
HTML 4.01 estricto Permite todos los elementos y atributos HTML, pero no permite etiquetas de presentación, como la etiqueta de fuente. No se permiten conjuntos de marcos.
HTML 4.01 conjunto de marcos Igual al HTML transicional, pero permite conjuntos de marcos.
XHTML 1.0 estricto Igual al HTML estricto, pero todo el contenido debe estar escrito como XML bien formateado. Por ejemplo, todas las etiquetas de apertura deben tener etiquetas de cierre. No se permiten conjuntos de marcos.
XHTML 1.0 transicional Igual al HTML transicional, pero todo el contenido debe estar escrito como XML bien formateado. No se permiten conjuntos de marcos.
XHTML 1.0 conjunto de marcos Igual al XHTML transicional, pero permite conjuntos de marcos.
XHTML 1.1 Igual al XHTML estricto, pero también proporciona capacidades para módulos, como soporte Ruby para idiomas orientales-asiáticos.

La declaración de doctype debe ser el primer elemento en un documento HTML, antes que html.

Precarga de archivos


Cuando se desea que un archivo se almacene en segundo plano se puede ocupar el atributo preload . A este atributo le pueden ser asignados tres valores:

none : no hay almacenamiento. Recomendado cuando una pagina tiene muchos archivos.


auto : hay almacenamiento automatica. Recomendado cuando la pagina tiene pocos archivos


metadata : solo hay almacenamiento de metadados de los archivos.


< audio src = "archivo.mp3" preload = "none/auto/matadata" > < /audio >


Formato Multiple

< audio controls >

INDEPENDENCIA DE PLUGINS


Para los archivos de audio y video se solía ocupar un plugin Flash en las páginas web para poder reproducir estos archivos. Se presentaban fallos constantemente y había que cerrar el navegador y volverlo abrir para que funcionara. Ahora, con las nuevas implementaciones, este plugin ya no se utiliza.
Al reproducir estos archivos a través de HTML5 se obtienen ciertas ventajas:

  1. Consume menos recursos
  2. Mejor rendimiento: tanto en la apertura como la reproducción.
  3. Versátil: funciona con videos incrustados en otras páginas web.

Interfaz Application Caché o Caché de aplicaciones.


Esta Tecnología sin conexion se caracteriza por:


  1. Mejorar el rendimiento de una aplicación
  2. Conservar datos de sesiones de usuario
  3. Exploración del sitio web sin conexión
  4. Descaga de recursos del servidor que han cambiado

Para conseguir esto, una aplicación requiere del famoso archivo Manifiesto donde se indica al navegador los ficheros que debe almacenar en el cache local. esto permite acceder a la página al no tener conexión disponible.
Este archivo en HTML luce de la siguiente manera:

< !DOCTYPE HTML >
< html lang = "en" manifest = "/ejemplo.appcache" >
< !- - lista de archivos que se traen de la página web - ->
< /html >


Para almacenar las páginas de la aplicación en caché, el atributo manifest debe incluirse en dichas páginas. Además de los ficheros la misma página que los contiene es almacenada. Si no se indica este atributo, el navegador simplemente no la almacena, pero si la página esta explicita en el archivo manifiesto, sí.
El atributo puede apuntar a una URL absoluta (incluye todas las partes de la url o relativa (contiene partes de una url para hacerla más breve), pero las URL absolutas deben tener el mismo origen que la aplicación web. El archivo como tal puede llevar cualquier extensión:


< !- - UTL Absoluta - - >
< html manifest = "http://www.ejemplo.com/ejemplo.mf" >
......
< /html >


El tipo MIME (Extensiones Multipropósito de correo de internet) para los archivos de manifiesto es text/cache-manifest. y para los archivos textuales es el valor por defecto. Un archivo de este tipo no puede contener binarios y debe ser legible.

LocalStorage o Almacenamiento Local.


Para comprender el concepto, primero se debe comprender el concepto de cookies Una cookie es un archivo generado por un sitio web que contiene fragmentos de datos que son enviados entre un emisor y un receptor. Básicamente, permiten identificar el historial de actividad de un usuario en un sitio web específico.
Ahora bien, las cookies presentan ciertos inconvenientes:

  1. Espacio limitado: Una cookie sólo puede ocupar 4kb de espacio. Por tal motivo, se ocupan para almacenar un identificador y este ser utilizado por el servidor para capturar la visita.
  2. Al realizar una petición al servidor, toda la información almacenada en las cookies se envía y recibe nuevamente con la respuesta del servidor. Por ende, los intercambios de información entre navegador web y servidor las cookies van anexas.
  3. Tienen una tiempo de vida.

En este punto entra el Almacenamiento Local , cuenta con las siguientes ventajas ante las cookies:


  1. Espacio menos limitado: de acuerdo al navegador puede ocupar entre 5MB y 10MB. Con este espacio se puede tener mas información.
  2. En las peticiones, la información almacenada NO es enviada al servidores.
  3. No existe una tiempo de vida, la información mantendra almacenada aunque se cierre el navegador, hasta que se elimine expresamente.

Para tener en cuenta. Mantener la información por un tiempo, no siempre es una buena idea. A veces lo que interesa es que la información se elimina una vez se cierre el navegador. Para este caso en particular, se debe usar sessionStorage.
SessionStorage cuenta con las mismas características que localStorage, pero con la diferencia que al cerrar el navegador se pierde la información.
HTML5 cuenta con una API en JavaScript para interactuar, si se desea se puede buscar más información al respecto de esta excelente herramienta.

Indexed Database y Web SQL


Indexed Database


Es una API de bajo nivel que ofrece almacenamiento en el cliente de cantidades significativas de datos. Para búsquedas de alto rendimiento en esos datos usa índices.
IndexedDB provee APIs separados para un acceso síncrono o asíncrono. El API síncrono está destinado a ser usado únicamente dentro de Web Workers, pero no será implementado aún por cualquier navegador. El API asíncrono trabaja con o sin Web Workers.

Web SQL


En ciertas ocasiones se requiere almacenar grandes cantidades de objetos localmente para satisfacer los requisitos de datos fuera de línea de las aplicaciones web. Web Storage o Almacenamiento Web es útil para almacenar pares de claves y sus valores. Sin embargo, no proporciona la recuperación en orden de claves, la búsqueda eficiente de valores o el almacenamiento de valores duplicados para una clave.
Esta especificación proporciona una API concreta para realizar una gestión de datos clave-valor avanzada. Se realiza ocupando de bases de datos transaccionales para almacenar claves y cada uno de sus valores (una o más por clave), y proporciona un medio para atravesar claves en un orden especifico. Se implementa mediante el uso de estructuras de datos de B-tree persistentes que se consideran eficientes para la inserción y la eliminación, así como en el cruce de un gran número de registros de datos.
