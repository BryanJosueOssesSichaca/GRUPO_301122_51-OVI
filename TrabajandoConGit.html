Flujo de Trabajo

En GIT existen 3 diferentes estados de trabajo

1 Working directory

Para editar, agregar archivos y trabajar en el proyecto.

2 Staging área

Para escoger los archivos están listos y los que no para pasar al tercer estado.

3 repository

Para llevar todo el registro en el proyecto (una vez hecho el commit).

Por lo tanto (1) siempre se tendrá un directorio donde se agregarán o modificarán archivos, una vez que la modificación de estos archivos este lista se seleccionan los archivos que están listos y se colocan en la staging area, de esta manera se indican los cambios que se desean incluir en el (3) commit que llevará registro de los cambios en el proyecto.

Preparar archivos para confirmación

Agregar archivos al staging área (Git Bash)

Una vez que se han producido cambios en los archivos existentes o se han agregado nuevos archivos al directorio de trabajo, al utilizar el comando $ git status se visualizan los archivos que han sido modificados o agregados, así como los cambios que están listos para confirmar y los que no.

Git Bash - comando status (07 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

Si se desea ver con exactitud, las líneas de código agregadas o eliminadas en estos cambios se utiliza el comando diff:

$ git diff

Si se desea ver únicamente los cambios sobre los archivos que están en el staging área o que están preparados para confirmación se agrega el parámetro --staged

$ git diff --staged

Una vez verificados los cambios realizados, para hacer commit sobre estos, primero se añaden al staging área, para eso se utiliza el comando add:

$ git add nombre_del_archivo

Al crear o modificar múltiples archivos, se pueden agregar estos cambios de una sola vez al staging área agregando el parámetro --A o --all

$ git add --all

NOTA: para quitar un archivo del staging area se utiliza el comando reset:

$ git reset HEAD nombre_del_archivo

Recordar que los archivos que son agregados en el commit (Confirmación) son aquellos que están en el staging área y han quedado registrados no aquellos archivos que solo han sido modificados.

Confirmando Cambios

Confirmando cambios (Git Bash)

Para confirmar los cambios se utiliza el comando commit (Confirmación):

$ git commit --m "Descripción del commit"

El parámetro --m permite escribir directamente la descripción del commit en la linea de comandos o terminal, en caso de no agregar el parámetro --m se abrirá el editor configurado para trabajar con GIT o el editor por defecto del sistema con el fin de escribir una descripción para el commit o confirmación.

Commit Git bash (07 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

En caso de querer modificar el mensaje de la descripción del commit se utiliza el parámetro --amend

$ git commit --ammend

Al estar en una rama local se puede deshacer el commit o confirmación realizada utilizando el comando reset

$ git reset --soft HEAD ~1

Al agregar el parámetro --soft se elimina el commit pero los archivos permanecen intactos y en caso de querer borrar todo lo que contenga el commit se utiliza el parámetro --hard

HEAD (cabeza) representa al commit actual, normalmente es el ultimo commit realizado , ~1 es la cantidad de commits que se retrocederán con respecto a HEAD.

Si se desea Deshacer un commit que se haya subido a una rama remota se utiliza el comando revert

$ git revert HEAD

Se puede sustituir HEAD por el código SHA del commit, puede remitirse al artículo crear y clonar repositorios con Git, donde se explica el procedimiento para ver el historial de commits utilizando el comando log, que imprime en pantalla entre otros datos el código SHA de confirmación del commit.

Confirmando cambios (Github desktop)

En el panel principal de Github Desktop, en la pestaña changes se aprecian dos campos de texto, el primero es llamado summary y el segundo mas grande description. En summary se debe escribir el nombre del commit y en description la descripcion del commit.

Por último se da clic en el botón commit to "Master" (Master hace referencia a la rama maestra o principal del repositorio, más adelante en este mismo artículo se hablará sobre las ramas).

Ramas o bifurcaciones

Una rama representa una linea del tiempo en el proyecto, son caminos independientes que pueden evolucionar de manera distinta, en la práctica representan diferentes estados que el proyecto puede tener.

Los cambios realizados en una rama solo se reflejan en la misma, esto permite experimentar o hacer grandes cambios en una rama diferente sin afectar la rama principal Master u otras ramas dentro del repositorio Git.

Al crear un repositorio, Git creará automaticamente la rama Master, es la rama por defecto y a partir de esta se crean nuevas ramas o bifurcaciones.

Ramas (Git Bash)

Si se desean ver las ramas que posee el repositorio se utiliza el comando branch:

$ git branch

Si se desea crear una nueva rama en el proyecto se indica como parámetro al comando branch el nombre de la rama.

$ git branch nombre_Rama

Git bash, creando nueva rama (07 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

Para cambiar entre ramas se utiliza el comando checkout:

$ git checkout nombre_Rama

Git bash, moverse entre ramas (07 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

Una vez alterada la rama, los cambios se realizarán sobre esta y no sobre master, si se desea reflejar los cambios realizados sobre esta nueva rama se deben fusionar(merge).

Al hacer merge entre dos ramas se generará un nuevo commit con los cambios derivados de la fusión de esas dos ramas.

Para fusionar ramas se utiliza el comando merge

$ git merge nombre_Rama

Git bash, mezclando ramas (07 de Abril del 2018) “elaboración propia - Bryan Josue Osses Sichaca”

Primero se ubica sobre la rama que absorberá a la otra (se utiliza el comando checkout para moverse entre ramas), en el ejemplo se sitúa sobre la rama Master para poder absorber en Master la otra rama llamada nueva_rama.

Si se desea eliminar una rama local se indica el parámetro -D al comando branch

$ git branch -D nombre_Rama
